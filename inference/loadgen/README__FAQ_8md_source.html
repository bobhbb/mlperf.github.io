3 ## Q: The LoadGen does not match the MLPerf specification. Who is right?
4 **A:**
5 The MLPerf spec is *always* right.
6 Please file a LoadGen bug so it may be resolved.
7 
8 ## Q: How can I file a bug?
9 **A:**
10 On GitHub: https://github.com/mlperf/inference/issues/new
11 
12 ## Q: Can I make local modifications to the LoadGen for submission?
13 **A:**
14 No. To keep the playing field level, please upstream any local
15 modificiations you need to make. Ideally upstream such changes behind a runtime
16 flag or via an abstract interface the client can implement. This will help
17 with testability.
18 
19 ## Q: Where can I find the results of a test?
20 **A:**
21 By default, the loadgen will output an *mlperf_log_summary.txt* file
22 that summarizes the target metrics and constraints of the test, along with
23 other stats about the run.
24 
25 *Note:* LogSettings also has a flag to forward the results to stdout and
26 there's an outstanding TODO to make this more programmable.
27 
28 ## Q: The reference implementation for <*some_model*> prints out results of its own. Are those for submission?
29 **A:**
30 They are not. The LoadGen results are the ground truth for submission
31 results since they will work even for systems that forgo the python bindings.
32 If you notice a bug in the LoadGen's results, please file a bug or submit a
33 patch.
34 
35 ## Q: I'm getting linker errors for LoadgenVersion definitions. Where is *version_generated.cc*?
36 **A:**
37 If you have a custom build setup, make sure you run the *version_generator.py*
38 script, which will create the cc file you are looking for. The official build
39 files that come with the LoadGen do this for you out of the box.
40 
41 ## Q: What is this *version_generator.py* script?
42 **A:**
43 The LoadGen records git stats (if available) and the SHA1 of all its
44 source files (always) at build time for verification purposes. This is easy
45 to circumvent, but try your best to run *version_generator.py* correctly;
46 ideally integrated with your build system if you have a custom build.
47 The intention is more to help with debugging efforts and detect accidental
48 version missmatches than to detect bad actors.

50 ## Q: How do I view the *mlperf_log_trace.json* file?
51 **A:**
52 This file uses the [Trace Event Format]
53 (https://docs.google.com/document/d/1CvAClvFfyA5R-PhYUmn5OOQtYMH4h6I0nSsKchNAySU/edit)
54 to record a timeline of all the threads involved.
55 You can view the file by typing [chrome://tracing](chrome://tracing) into
56 Chrome's address bar and dragging the json file there.
57 This file zips well and you can drag the zip file directly into
58 [chrome://tracing](chrome://tracing) too.
59 Please include zipped traces (and the other logs) when filing bug reports.
60 
61 ## Q: What is the difference between the MultiStream and MultiStreamFree scenarios?
62 **A:**
63 MultiStream corresponds to the official MLPerf scenario for submissions;
64 it has a fixed query rate and allows only one outstanding query at a time.
65 MultiStreamFree is implemented for evaluation purposes only; it sends queries
66 as fast as possible and allows up to N outstanding queries at a time. You may
67 want to use MultiStreamFree for development purposes since small improvements
68 in performance will always be reflected in the results, whereas MultiStream's
69 results will be quantized.
70 
71 ## Q: Why is the code littered with so many lambdas? My eyes hurt.
72 **A:**
73 Lambdas are a convenient and efficient way to ship arbitrary data + deferred
74 logic over to the logging thread without much boilerplate.
75 Much of the loadgen is built on top of the logging utilities.
76 Thus the lambdas. (Sorry about the eyes.)
77 
78 ## Q: What C++ version does the LoadGen target?
79 **A:**
80 It currently targets and requires C++14. It should compile with recent
81 versions of clang, gcc, and msvc.
82 
83 ## Q: What dependencies does the LoadGen code have?
84 **A:**
85 The C++ code has no external dependencies. The loadgen itself, logging
86 utilities, and unit test utilities are built solely on the C++ Standard Library.
87 The python bindings, however, do require
88 [pybind11](https://github.com/pybind/pybind11).
