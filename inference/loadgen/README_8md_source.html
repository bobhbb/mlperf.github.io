1 # Overview {#mainpage}
2 
3 * The LoadGen is a *reusable* module that *efficiently* and *fairly* measures
4  the performance of inference systems.
5 * It generates traffic for scenarios as formulated by a diverse set of experts
6  in the [MLPerf working group](https://mlperf.org/about).
7 * The scenarios emulate the types of workloads seen in mobile devices,
8  autonomous vehicles, robotics, and cloud-based setups.
9 * Although the LoadGen is not model or dataset aware, its strength is in its
10  reusability with logic that is.

24 ## Scope of the LoadGen's Responsibilities
25 
26 ### In Scope
27 * **Provide a reusable** C++ library with python bindings.
28 * **Implement** the traffic patterns of the MLPerf inference scenarios and
29  modes.
30 * **Record** all traffic generated and received for later analysis and
31  verification.
32 * **Summarize** the results and whether the scenario's performance constraints
33  were met.
34 * **Target high-performance** systems with an efficient multi-thread friendly
35  logging infrastructure.
36 * **Generate trust** via a well-tested and community-hardened code base.
37 
38 ### Out of Scope
39 * **It is not** aware of the ML model it is running against.
40 * **It is not** aware of the data formats of the model's inputs and outputs.
41 * **It is not** aware of how to score the acuracy of a model's outputs.
42 * **It is not** aware of MLPerf rules regarding scenario-specific constraints.
43 
44 Limitting the scope of the LoadGen in this way keeps it reusable across
45 different models and datasets without modification. Using composition and
46 dependency injection, the user can define their own model, datasets, and
47 metrics.
48 
49 Additionally, not hardcoding MLPerf-specific test constraints, like test
50 duration and performance targets, allows users to use the LoadGen unmodified
51 for custom testing and continuous integration purposes.

53 ## Submission Considerations
54 
55 ### Upstream all local modifications
56 * As a rule, no local modifications to the LoadGen's C++ library are allowed
57 for submission.
58 * Please upstream early and often to keep the playing field level.
59 
60 ### Choose your TestSettings carefully!
61 * Since the LoadGen is oblivious to the model, it can't enforce the MLPerf
62 requirements for submission. *e.g.:* target percentiles and latencies.
63 * For verification, the values in TestSettings are logged.
64 * *Note:* There is an effort to load TestSettings from a config file to make
65 it easier to respect submission requirements, but it is a work in progress.

67 ## Responsibilities of a LoadGen User
68 
69 There are numerous eamples in demos, tests, and reference models of code
70 that integrates with the LoadGen that provide detailed templates for you to
71 follow. At a high-level, to use the LoadGen, a user must:
72 
73 * Implement the Iterfaces
74  + Implement the SystemUnderTest and QuerySampleLibrary interfaces and pass
75  them to the StartTest function.
76  + Call QuerySampleComplete for every sample received in IssueQuery.
77 
78 * Assess Accuracy
79  + Process the mlperf_accuracy_log.json output by the LoadGen to determine
80  the accuracy of your system.
81  + Python scripts will be provided by MLPerf model owners for you to do
82  this automatically for the official models.
